## 模块一：Context / Performance

### 1. 首轮长窗口或多模态输入时 first token 很慢，怎么把体感从 5-10 秒压到 2 秒？

我会先把问题拆成两层：一层是真实模型开始输出的 TTFT，另一层是用户感知到系统有反馈的时间。长窗口和多模态输入慢，主要慢在 prefill，也就是模型要先把长文本、图片、PDF、工具 schema、系统提示都编码完，才能开始生成。所以如果完全等大模型处理完再吐第一个 token，成本会很高，也很难稳定到 2 秒。

我的方案是做“快速响应 + 后台预处理 + 缓存复用”。

第一步，前端或服务端在 1-2 秒内先给用户一个轻量反馈。这个反馈不一定由主模型生成，可以是模板、小模型或快速分类模型生成，比如：“我已经收到材料，先解析图片和长文档，马上给你结论。”这样用户知道系统没有卡住。这个回答不能假装已经完成推理，只表示正在处理。

第二步，把重输入尽量前置处理。用户上传文件或图片时，后台立即做 OCR、图片 caption、PDF 分页、表格抽取、chunk、embedding、缩略图、关键信息摘要。等用户真正提问时，不把完整原文全部塞进上下文，而是先根据问题取相关片段、相关页、相关图片区域，再交给主模型。

第三步，稳定 prompt 前缀。系统提示、工具定义、用户长期偏好、项目背景、文档摘要这些尽量放在固定位置，变量问题放在最后。这样服务端或模型侧更容易命中缓存，减少重复 prefill。对于多轮对话，尤其要避免每轮都重排工具 schema 和系统提示，否则缓存命中率会变差。

第四步，拆模型路径。首轮先用轻模型判断任务类型、是否需要检索、是否需要视觉模型。如果只是问“总结这个文档”，可以先给目录级摘要；如果问的是文档中某个具体细节，就只取相关 chunk。真正需要全量理解时，再进入重模型。

整体链路是：

```text
upload/input -> async preprocess -> cache/index -> query router
             -> fast ack within 2s -> main model stream answer
```

这个设计的重点不是骗用户，而是把“系统无响应”的时间变短。真正的重推理如果必须处理几十万 token 或大量图片，不一定每次都能真实 TTFT 到 2 秒，但用户体感可以稳定控制在 2 秒左右。

### 2. 一个 session 连续聊了 200 轮，context 快爆了，怎么压缩并保证流畅？

我不会等 context 真的爆了才压缩，而是设置阈值，比如到 70%-80% 就开始做增量压缩。压缩分三层：

第一层是最近上下文，保留最近 10-20 轮原文。最近对话决定语气、指代关系和当前操作状态，不能过早摘要。

第二层是结构化摘要，把更早的内容压成固定 schema，而不是随便总结一段自然语言。我会保留：

```text
用户最终目标：
当前任务状态：
已经完成的步骤：
还没完成的步骤：
关键约束：
用户偏好：
重要实体、文件、链接、账号、环境：
关键决策和原因：
工具调用结果：
上次停在哪里：
待确认问题：
不能重复做的事情：
```

第三层是外部记忆和冷存储。稳定偏好、长期项目事实进入 memory；完整 transcript、工具输出、文件 diff、日志进入冷存储，后面需要时用检索召回，而不是一直占上下文。

为了保证压缩后对话仍然流畅，我会做几个校验：

1. 压缩摘要必须能回答三个问题：用户现在要什么、我下一步该做什么、哪些约束不能忘。
2. 保留关键原话，尤其是用户明确说过的偏好、否定项、最终决定。
3. 保留最近原文，避免“他/这个/刚才那个”这种指代断掉。
4. 压缩后让模型做一次 continuity check，如果发现目标、约束、任务状态缺失，就重新压缩。
5. 摘要版本化，保留被压缩事件的 id，后面如果出现争议，可以回查原始记录。

我会把压缩看成 agent 的状态迁移，而不是单纯省 token。压缩后的上下文要能继续执行任务，而不是只让人看懂发生了什么。

## 模块二：Memory

### 1. 用户半个月后问了以前问过的问题，Agent 怎么召回 memory 更合理？

我不会一上来就把以前的答案原样贴出来。更合理的流程是：先判断这个问题是否需要 memory，再召回，再校验，再融合回答。

第一步是触发判断。不是所有问题都要查 memory，比如普通常识问题没必要。但如果问题涉及用户偏好、长期项目、历史决策、重复问法、个人背景、之前做过的方案，就应该触发 memory recall。

第二步是混合召回。只靠向量相似度不够，因为用户半个月后可能换了说法。我会同时用语义向量、关键词、实体、时间、会话主题、任务 id 做召回，然后 rerank。

第三步是可信度判断。召回结果要分类型：

- 用户明确说过的偏好，可信度高。
- 当时某个任务的结论，需要看是否过期。
- 外部实时信息，比如价格、招聘状态、接口文档，可能已经变了，不能直接复用。
- 模型自己推断出来的东西，可信度低，需要谨慎。

第四步是回答融合。比较自然的说法是：

```text
你之前问过类似问题，当时我们定下来的结论是 X。
如果沿用这个结论，这次我建议还是 Y。
但这里有一个可能变化的点是 Z，所以如果要做最终决策，我会重新确认它。
```

这样用户会感觉 Agent 记得之前的上下文，但不会因为旧记忆过期而给出错误答案。

### 2. 我理解的 Agent memory 经典框架和发展趋势

我理解的 Agent memory 可以分成五类。

第一类是 working memory，也就是当前上下文、当前任务状态、当前计划。它生命周期短，但对当前执行最重要。

第二类是 episodic memory，也就是过去发生过的具体事件，比如某次对话、某个工具调用、某次用户纠正、某个任务结果。

第三类是 semantic memory，也就是稳定知识，比如用户偏好、项目架构、团队规范、长期目标、常用环境。

第四类是 procedural memory，也就是“怎么做事”的记忆，比如用户喜欢先给结论再解释、某个项目测试命令怎么跑、某类任务应该先查哪些文件。

第五类是 reflection 或 consolidation，也就是把多次经历归纳成更稳定的规则。比如用户多次纠正“不要写得太像 AI”，系统就应该总结成一个长期写作偏好。

经典链路是：

```text
observe -> extract -> store -> retrieve -> rerank -> inject -> update/forget
```

这里最难的不是存，而是判断什么值得存、什么时候该召回、召回后怎么不打扰用户。

我觉得趋势有几个：

1. 从手动保存变成自动综合。用户不想每次说“记住这个”，系统应该从长期交互中自动归纳。
2. 从单条记忆变成 memory graph。人、项目、文件、偏好、任务之间有关系，不只是一个向量库。
3. 从“全都塞进 prompt”变成按需召回。不同任务加载不同记忆，避免污染上下文。
4. 从黑盒个性化变成可解释、可编辑。用户应该知道这次回答用了哪些记忆，也能删除和纠正。
5. 从聊天记忆扩展到工作空间记忆。未来 memory 不只来自聊天，还会来自文件、邮件、日历、代码仓库、工具日志。

头部产品大概分两条路线：通用聊天产品更重视用户偏好、跨会话个性化和隐私控制；coding agent 更重视项目规则、构建命令、调试经验、文件路径和任务状态。

## 模块三：Task

### 1. 长程任务中模型执行一段时间后忘掉目标，有哪些解决方案和优缺点？

长程任务忘目标，核心原因是目标只存在自然语言上下文里，而不是被 runtime 当成一等状态管理。我会用几种方式组合解决。

第一种是显式 task state。把目标、验收标准、当前步骤、已完成、未完成、阻塞项写成结构化状态，每轮都注入。优点是稳定，缺点是要维护状态一致性，如果模型错误更新状态，会把错误固化。

第二种是 todo/checklist 工具。模型每完成一步都更新 checklist，用户也能看到进度。优点是可视化强、适合长任务；缺点是模型可能机械打勾，需要验证机制。

第三种是周期性 self-check。每执行几步就让模型回答：“当前目标是什么？我是否偏离？下一步是什么？”优点是简单，缺点是增加 token 和延迟。

第四种是 event log + checkpoint。所有用户输入、工具调用、工具结果、状态更新都写成事件。任务中断或上下文压缩后，可以从 checkpoint 恢复。优点是可靠、可追溯；缺点是工程复杂。

第五种是 critic/verifier。让另一个模型或规则系统检查当前行动是否符合目标和验收标准。优点是质量高，缺点是成本更高，而且 verifier 本身也可能误判。

我的实际方案会是：

```text
goal spec + task state + checklist + event log + periodic verifier
```

其中 goal spec 和 checklist 面向用户，event log 和 checkpoint 面向 runtime，verifier 面向质量控制。

### 2. 用户要求每天早上 9 点根据昨天聊天情况做复盘总结，怎么设计？

我会把它设计成一个自动化任务系统，而不是普通聊天提醒。

用户创建任务时，需要保存：

```text
automation_id
user_id
timezone
schedule: 每天 09:00
summary_scope: 昨天 00:00-23:59 的聊天
data_sources: 哪些会话、哪些项目、是否包括文件和工具结果
delivery_channel: 站内、邮件、微信、Slack
prompt_template
privacy_policy
enabled/disabled
```

每天 9 点触发后，worker 会按用户时区计算“昨天”的时间范围，然后拉取聊天事件。拉取后先做过滤：临时会话不进总结，用户排除的会话不进总结，敏感内容要么跳过，要么只做本地摘要，不进入外部模型。

然后做聚类和总结：

```text
昨天主要聊了什么：
完成了哪些事：
形成了哪些结论：
有哪些未完成事项：
今天建议优先做什么：
有哪些需要用户确认的问题：
```

如果昨天聊天很多，我会先按主题分组做局部摘要，再汇总成日报。如果昨天几乎没聊天，就发一条很短的总结，不强行生成内容。

工程上要注意幂等性。比如 `daily_summary:{user_id}:{date}` 必须唯一，防止 worker 重试导致重复发送。失败要有重试，连续失败要告警。用户必须能暂停、修改时间、修改总结范围、删除历史总结。

## 模块四：Tool / Session Runtime

### 1. 工具有同步和异步两类，异步工具怎么执行和通知？

我会统一抽象成 tool job。

同步工具适合短任务，比如读一个配置、查一个字段、计算一个表达式。它在当前模型回合内执行，直接返回 `tool_result`。

异步工具适合长任务，比如爬取网页、生成报告、跑测试、训练模型、调用第三方慢接口。它不能让用户一直等，所以模型发起调用后，runtime 立即创建 job，并返回：

```text
job_id
tool_name
status: queued/running
estimated_time
progress_url 或 progress event
```

后台 worker 执行工具，执行过程不断写事件：

```text
ToolStartedEvent
ToolProgressEvent
ToolCompletedEvent
ToolFailedEvent
ToolCancelledEvent
```

工具完成后有两种情况：

如果用户还在当前 session，并且这个结果对当前回答重要，就唤醒 agent，让模型基于结果继续生成最终答复。

如果用户已经离开，或者结果只是通知型，就发站内通知、邮件、webhook，用户点开后能回到对应 session。

异步工具一定要有 idempotency key，防止重复执行；要有 timeout 和 cancel；要有权限校验，尤其是发邮件、下单、改数据库这类副作用工具；结果要结构化保存，不能只放一段自然语言。

### 2. session state 为 busy 时，用户又发消息或异步工具完成，runtime 怎么处理？

我不会让 busy 变成一个大锁，把所有事件都挡住。我的做法是 append-only event queue：任何新消息、工具完成、取消请求都先写事件，再由 session actor 串行处理。

如果用户在 busy 时发普通补充消息，比如“顺便也考虑成本”，runtime 先把它写成 `UserMessageEvent`，标记为 pending。当前模型回合如果还没进入关键动作，可以尝试 interrupt 并重启；如果已经在工具执行或提交阶段，就等当前安全点结束后再合并处理。

如果用户发的是“停下”“取消”“换个目标”，这类消息优先级更高。runtime 应该中断当前 run，取消可取消的工具 job，然后基于新用户消息重新规划。

如果异步工具完成事件到了，也先写入事件流。如果当前 run 正在等待这个 job，就立即唤醒；如果模型正在处理另一个回合，就不要强行把工具结果插进去，而是在下一次 reducer 构造上下文时加入。

我会把状态拆成多个维度，而不是只用一个 busy：

```text
run_state: idle/running/paused/cancelling/error
tool_state: no_job/waiting/running/completed/failed
input_state: no_pending/pending_user_input/pending_interrupt
notification_state: silent/notify_on_complete
```

这样用户插话、工具完成、模型生成、暂停恢复都能有明确规则。核心原则是：事件可以并发到达，但状态转换必须串行；用户的取消和修改目标优先级最高。

## 模块五：Agent Runtime 架构对比

### 1. Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？优缺点是什么？

我理解它们最大的区别是：Claude Code 更像是为 agent runtime 设计的事件式工具交互，而 OpenAI-compatible function calling 更像是通用聊天 API 里的函数调用协议。

Claude/Claude Code 的工具使用通常是模型输出一个结构化的 tool use block，runtime 执行工具后，再把 tool result 作为对应结果喂回模型。它还会把 bash、文件编辑、搜索、电脑操作、MCP 等能力都纳入 agent loop。对于 coding agent 来说，这种方式比较自然，因为工具调用不是简单函数，而是一次环境交互：可能有 stdout、stderr、文件 diff、截图、权限确认、失败重试。

GLM、豆包这类 OpenAI-compatible function calling 通常采用类似 Chat Completions 的结构：请求里传 `tools`，模型返回 `tool_calls`，应用侧执行函数，再用 `role=tool` 和 `tool_call_id` 把结果返回。它的好处是标准化程度高，接入生态好，迁移模型比较容易。很多业务系统只要适配一次 tools schema，就能在多个模型之间切换。

Claude 这种方式的优点是：

- 更适合长程 agent。
- 工具结果可以和 agent 事件流结合。
- 对代码编辑、终端、文件系统、多步骤任务更自然。
- 更容易做权限确认、暂停恢复、观察日志。

缺点是：

- 厂商绑定更强。
- runtime 设计更复杂。
- 如果业务只是普通函数调用，会显得重。

OpenAI-compatible function calling 的优点是：

- 标准化，容易接入。
- 业务函数、API 查询、数据库查询这类场景很方便。
- 多模型兼容成本低。
- 工具 schema 和调用结果都比较直观。

缺点是：

- 很多复杂工具结果最后会被压成 JSON 或文本 blob。
- 对长程 agent 的状态、权限、并发、恢复表达不够完整。
- 不同厂商虽然都说 compatible，但在 streaming、parallel tools、reasoning content、strict schema 上经常有细节差异。

所以我的判断是：如果做普通业务助手，OpenAI-compatible function calling 更轻；如果做 coding agent、桌面 agent、长程任务 agent，Claude Code 这类事件式 runtime 更适合。

### 2. OpenHands 的状态机设计有什么优缺点？更优雅的实现方式是什么？

OpenHands 这类项目的价值在于，它不是只把 agent 看成一个“模型请求”，而是把用户消息、模型动作、工具观察、错误、暂停、压缩都当成事件。这个方向是对的，因为真实 agent 会遇到长任务、工具失败、用户插话、上下文压缩、并行工具和恢复执行。

如果用传统状态机，比如：

```text
INIT -> RUNNING -> WAITING_TOOL -> PAUSED -> ERROR -> FINISHED
```

优点是直观，UI 好展示，新人也容易理解。小规模任务下，这种状态机很好用。

但缺点也明显：一旦加入异步工具、用户中途打断、多个工具并行、上下文压缩、人工确认、失败重试，状态会越来越多。最后会变成：

```text
RUNNING_BUT_WAITING_TOOL_AND_HAS_PENDING_USER_INPUT_AND_NEED_COMPACTION
```

这说明单一状态机已经装不下真实 runtime 复杂度。

更优雅的实现方式是 event-sourced actor。

每个 session 是一个 actor。所有输入都是事件：

```text
UserMessageEvent
AssistantMessageEvent
ToolCallEvent
ToolResultEvent
ToolErrorEvent
PauseEvent
ResumeEvent
CancelEvent
CompactionEvent
StateUpdateEvent
```

事件只追加，不修改。当前状态不是手写一堆 if-else，而是由 reducer 从事件流推导出来。agent loop 只做一件事：读取当前 derived state，决定下一步 command，再产生新事件。

我会把状态拆成几个正交维度：

```text
run lifecycle: idle/running/paused/cancelling/error/done
tool jobs: queued/running/completed/failed/cancelled
context budget: normal/near_limit/compacting/compacted
user input: none/pending/interrupt
delivery: active/background/notify_on_complete
```

这样比一个巨大的 enum 更稳定。它的优势是：

- 可以回放，方便 debug。
- 可以恢复，中断后从事件流重建状态。
- 可以审计，每次工具调用和状态变化都有记录。
- 可以处理并发事件，因为事件入队后由 actor 串行消费。
- 可以自然接入压缩，压缩只是事件流中的一个 condensation event。

我认为更好的 agent runtime 不应该只追求“状态机画得简单”，而应该追求事件模型清晰、状态可推导、任务可恢复、工具可审计。状态机仍然可以用于 UI 展示，但底层应该是事件驱动。
